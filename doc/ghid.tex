\subsubsection{Folosind OpenOGEHL clasic}

\begin{verbatim}
	#!/bin/bash
	
	# Usage:
	#   ./gen                        - configure (cmake)
	#   ./gen b                      - build
	#   ./gen <benchmark> r          - run default config for benchmark
	#   ./gen <benchmark> c1..c9     - run named config for benchmark
	#   ./gen <benchmark> all        - run all configs for benchmark
	#
	# Benchmarks: applu | cc1 | mgrid | compress | ijpeg
	#
	# OGHEL predictor params: -bpred:oghel <num_tables> <table_size> <counter_len>
	# Defaults:               num_tables=7  table_size=2048  counter_len=4
	
	BASE_CMD="./build/sim-bpred -max:inst 500000000 -nice 1 -bpred oghel"
	
	# ---- benchmark selection ----
	BENCHMARK="${1}"
	case "$BENCHMARK" in
	applu)    SS_FILE="applu.ss";       IN_FILE="applu.in"   ;;
	cc1)      SS_FILE="cc1.ss";         IN_FILE="1stm.i"   ;;
	mgrid)    SS_FILE="mgrid.ss";       IN_FILE="mgrid.in"   ;;
	compress) SS_FILE="compress95.ss";  IN_FILE="bigtest.in" ;;
	ijpeg)    SS_FILE="ijpeg.ss";       IN_FILE="vgio.ppm"     ;;
	b)
	cmake --build build -j 12
	exit 0
	;;
	"")
	rm -rf build
	cmake -S . -B build -G "Unix Makefiles"
	exit 0
	;;
	*)
	echo "Unknown benchmark: $1"
	echo "Usage: $0 [applu|cc1|mgrid|compress|ijpeg] [r|c1..c9|all]"
	exit 1
	;;
	esac
	
	INPUT="./inputs/ss_files/${SS_FILE}"
	STDIN="./inputs/in_files/${IN_FILE}"
	mkdir -p "./results/${BENCHMARK}"
	
	run_config() {
		local tag="$1"
		local num_tables="$2"
		local table_size="$3"
		local counter_len="$4"
		echo "=== ${BENCHMARK}/${tag}: num_tables=$num_tables  table_size=$table_size  counter_len=$counter_len ==="
		$BASE_CMD \
		-bpred:oghel $num_tables $table_size $counter_len \
		-redir:sim  "./results/${BENCHMARK}/${tag}_simout.res" \
		-redir:prog "./results/${BENCHMARK}/${tag}_progout.res" \
		"$INPUT" < "$STDIN"
	}
	
	CMD="${2:-r}"
	
	case "$CMD" in
	r)   run_config default 7 2048 4 ;;
	# ---------- named configurations ----------
	#                       num_tables  table_size  counter_len
	c1)  run_config c1   4   512   3 ;;  # min tables, small table, short counters
	c2)  run_config c2   4  2048   4 ;;  # min tables, default size
	c3)  run_config c3   7   512   3 ;;  # default tables, small table, short counters
	c4)  run_config c4   7  2048   4 ;;  # default (baseline)
	c5)  run_config c5   7  2048   5 ;;  # default tables/size, long counters
	c6)  run_config c6   7  8192   4 ;;  # default tables, larger table
	c7)  run_config c7  12  2048   4 ;;  # max tables, default size
	c8)  run_config c8  12  8192   4 ;;  # max tables, larger table
	c9)  run_config c9  12  8192   5 ;;  # max tables, larger table, long counters
	all)
	for cfg in c1 c2 c3 c4 c5 c6 c7 c8 c9; do
	"$0" "$BENCHMARK" "$cfg"
	done
	;;
	*)
	echo "Unknown command: $CMD"
	echo "Usage: $0 [applu|cc1|mgrid|compress|ijpeg] [r|c1..c9|all]"
	exit 1
	;;
	esac
	elif [[ "$1" == "all" ]]; then
	for cfg in c1 c2 c3 c4 c5 c6 c7 c8 c9; do
	"$0" "$cfg"
	done
	
	else
	echo "Unknown option: $1"
	echo "Usage: $0 [b|r|c1..c9|all]"
	exit 1
	fi
	
\end{verbatim}

\subsection{Prezentarea predictorului OGhel}

Predictia ramificatiilor reprezinta una dintre tehnicile fundamentale de optimizare a performantei procesoarelor moderne cu pipeline adanc.
In absenta unei predictii corecte, procesorul este nevoit sa goleasca pipeline-ul si sa reia executia de la adresa corecta,
operatie cunoscuta sub numele de \textit{branch misprediction penalty}. De-a lungul timpului, cercetatorii au propus o
varietate de strategii cu complexitati si acurateti diferite.

\textbf{Predictori statici si bazati pe Markov.}
Cei mai simpli predictori sunt cei statici: \textit{always taken} si \textit{always not-taken}, care nu tin cont de istoricul executiei.
O treapta superioara o reprezinta predictorul bazat pe \textbf{lanturi Markov}, care modeleaza probabilitatea urmatorului rezultat
al unui salt in functie de cele $k$ rezultate anterioare ale aceluiasi salt. Desi intuitiv, modelul Markov este limitat de faptul
ca ia in considerare exclusiv istoricul local al unui singur salt, ignorand corelatiile cu alte ramificatii din program.

\textbf{Predictori globali -- familia GAg/GAp/PAp.}
Schema \textbf{GAg} (\textit{Global history, All branches, global prediction table}) introduce notiunea de
\textit{registru de istorie globala} (GHR -- \textit{Global History Register}), in care se acumuleaza
rezultatele ultimelor $W$ salturi, indiferent de adresa acestora. Indicele in tabela de predictie se
formeaza direct din GHR. \textbf{GAp} pastreaza istoricul global, dar foloseste tabele de predictie
per-ramificatie, reducand aliasul dintre intrari. \textbf{PAp} merge mai departe, combinand un
registru de istorie per-ramificatie cu tabele de predictie individuale. Aceste scheme ofera o
acuratete ridicata pentru programe cu corelatii inter-ramificatii, insa tabela creste exponential
cu adancimea istoricului.

\textbf{Predictorul GShare si predictori combinati.}
\textbf{GShare} aplica operatia XOR intre GHR si adresa PC a saltului pentru a indexa tabela de predictie,
reducand aliasul fara a creste semnificativ dimensiunea structurii. Predictorii \textbf{combinati} (de tip \textit{tournament}),
cum este cel implementat in SimpleSim sub optiunea \texttt{comb}, mentin doua predictori de baza si o \textit{meta-tabela}
care decide dinamic caruia sa ii acorde incredere pentru fiecare salt in parte.

\textbf{Predictori bazati pe perceptron.}
O directie moderna si importanta este reprezentata de predictorii bazati pe \textbf{perceptron}. Fiecare ramificatie este asociata
unui vector de ponderi; rezultatul predictiei este determinat de semnul produsului scalar dintre vectorul de ponderi si vectorul de
istorie globala (codificat ca $\pm 1$). Antrenamentul consta in incrementarea sau decrementarea ponderilor corespunzatoare in functie
de rezultatul real. Avantajul principal este capacitatea de a captura corelatii liniare pe istorii foarte lungi, la un cost de stocare
rezonabil. Limita fundamentala a perceptronului simplu consta insa in incapacitatea de a invata functii nelineare: de exemplu, paritatea
unui sir de biti (corelatii XOR) nu poate fi reprezentata de un model liniar.

\textbf{Predictorul O-GEHL.}
\textbf{O-GEHL} (\textit{O-Geometric History Length}) depaseste limitarile mentionate mai sus printr-o serie de mecanisme specifice.
In locul unei singure tabele indexate dupa un istoric de lungime fixa, O-GEHL utilizeaza \textbf{mai multe tabele independente} $T_0, T_1, \ldots, T_{M-1}$,
fiecare indexata dupa un istoric de lungime diferita. Lungimile istoricului formeaza o \textbf{progresie geometrica}: $L_i = \lfloor \alpha^i \cdot L_0 \rceil$,
unde $\alpha > 1$ este un factor de scala. Aceasta alegere asigura acoperirea uniforma a fenomenelor de corelatie la scari multiple ale
istoricului, de la corelatie pe termen scurt pana la corelatie pe termen lung. Predictia finala se obtine prin \textbf{suma algebrica}
a valorilor contoarelor saturate extrase din fiecare tabela -- un mecanism similar votului ponderat al unui ansamblu de perceptroni,
dar fara a fi limitat la corelatii liniare, datorita indexarii diferentiate. Aceasta suma este comparata cu zero: daca este pozitiva,
saltul este prezis ca efectuat (\textit{taken}), altfel ca neefectuat. Spre deosebire de predictori globali simpli, \textbf{aliasul}
este gestionat prin diversificarea functiilor de indexare: fiecare tabela foloseste o combinatie diferita de biti din PC si din GHR,
reducand probabilitatea ca doua ramificatii distincte sa partajeze acelasi contor in toate tabelele simultan. In concluzie, O-GEHL
reuneste avantajele istoricului global (capturarea corelatiilor inter-ramificatii), ale lungimilor geometrice (acoperire multi-scala),
si ale votului prin suma (robustete la zgomot si alias), oferind o acuratete superioara predictorilor clasici la un buget de hardware comparabil.

\subsection{Compiland OpenOGEHL}
Asa cum am promis in capitolul anterior, este timpul sa prezentam succint modul in care OpenOGEHL (fie in varianta ei clasica sau ca modul integrat) este compilat.\\

Avand in vedere ca proiectul implementat de noi si SimpleSim au fost scrise folosind limbajul de programare C, nu ne bucuram de facilitatea unui \textit{package manager}, adica a unui utilitar ce poate gestiona pachetele/modulele/bibliotecile folosite de aplicatia noastra si dependintele acestora. Un bun exemplu de astfel de utilitar este \textbf{cargo}, folosit in limbajul Rust. Acesta este responsabil de instalarea modulelor pe calculatorul programatorului si fixarea versiunilor acestora. Limbajul C in schimb, este mai vechi si a fost conceput a fi minimalist si simplu (structural vorbind, nu neaparat ca uz), astfel acesta se bazeaza pe manipularea directa a driver-ului GCC (GNU Compiler Collection) pentru a compila aplicatia. Preferam sa folosim termenul driver si nu compilator deoarece GCC intern alcatui din preprocesor (cpp), compilator (cc1), asamblor (as) si link-editor(ld), care practic corespond etapelor procesului de compilare a oricarui program scris in C.

	\includegraphics[width=0.7\textwidth]{images/gcc.jpg}

Totusi, pentru a face mai lesne acest proces de compilare a aplicatilor, programatorii tind sa foloseasca asa numitul \textit{build system} pe care l-am mentionat in capitolul anterior. In cazul limbajului C cele mai populare solutii sunt \textbf{Make} si \textbf{CMake}. In timp ce cel dintai sistem este mai degraba un procesor de text (flexibil si cu multe facilitati totusi), cel de al doilea reprezinta o solutie moderna aplicata in mai multe proiecte (cum ar fi de exemplu Raylib, biblioteca grafica folosita pentru OpenOGEHL clasic). Mecanismele si principiile prin care Make si CMake nu vor fi elaborate nefiind scopul acestei lucrari. \subsubsection{Arhitectura software OpenOGEHL clasic}
Pentru a asigura o buna calitate software a aplicatiei si potentiala extindere a acesteia in diferite directii, s-a pus problema intocmirii unei arhitecturi simple dar robuste. Vom discuta in cele ce urmeaza cum a fost aceasta realizata pentru aplicatia clasica de OpenOGEHL, aceste decizii fiind valabile intr-o masura sau alta si in momentul integrarii cu SimpleSim.\\

In ansamblu, aplicatia este constituita din urmatoarele componente principale: fisierele de tip trace ce conțin traseul parcurs de program, interfata grafica, responsabila pentru redarea utilizatorului a informatiilor necesare, preluarea configurarilor acestuia si transmiterea datelor necesare predictorului de salt (ex: numarul de tabele folosit), si intr-un final, predictorul de salturi in sine, ce implementeaza algoritmul O-GEHL discutat anterior, responsabila de a asemenea cu returnarea catre interfata grafica a ratei de miss (cat la suta din numarul prezis de salturi nu s-au efectuat de fapt)

\subsubsection{Arhitectura software OpenOGEHL integrat}
Fiind acum familiari despre cum a fost aplicatia proiectata initial, este momentul sa discutam despre masurile luate pentru integrarea acesteia. Daca este sa ne aducem aminte de arhitectura SimpleSim vom observa usor ca predictorul nostru de ramificatii este situat la nivelul cel mai de jos, adica la nivelul Performance Core (responsabil pentru redarea comportamentului executiei micro-arhitecturii procesorului).Astfel, componenta grafica implementata de noi nu ne mai este folositoare, caci SimpleSim ofera deja utilizatorului o anumita interfata cu care poate lucra. Nemaifiind nevoie de un GUI evident ca putem simplifica structura si mai mult prin eliminarea firelor de executie, ramanand astfel doar cu logica ce implementeaza comportamentul unui predictor tip O-GEHL. \\

	\centering
	\includegraphics[width=0.7\textwidth]{images/sim_drawio.png}
	\caption{OpenOGEHL ca parte a SimpleScalar}

Din perspectiva programarii, fiecare modul din SimpleScalar este alcatuit dintr-un fisier antet (*.h) ce defineste interfata disponibila utilizatorului si un fisier de implementare (*.c). Cu exceptia a catorva fisiere *.h OpenOGEHL nu se abata de la aceasta regula si poate fi folosit in program prin simpla includere a antetului \textit{bp.h}. In cazul acestui proiect modulul este inclus ca parte a \textit{bpred} conform figurei urmatoare:

\begin{figure}[H]
	\centering
	\includegraphics[width=0.7\textwidth]{images/arch2.png}
	\caption{Includerea OpenOGEHL la nivel de fisier}
\end{figure}

Aceasta includere nici macar nu este resimtita de catre programator deoarece din punctul sau de vedere el sau ea lucreaza doar cu componenta bpred. Evident, data fiind natura limbajului C in care nu exista o separare rigida intre interfata si implementare (cum ar fi de exemplu cazul limbajului C++), nimic nu impiedica utilizarea directa a OpenOGEHL a aplicatiei trecand astfel peste un nivel intermediar.

Asa ca, ca daca ar fi sa privim prin prisma formalismului interfata-implementare, observam ca urmatoarele functii sunt puse la dispozitie:

\begin{itemize}
	\item BP\_Init(): responsabila pentru initializarea tabelelor din predictor
	\item BP\_GetPrediction(): responsabila prezicerea saltului
	\item BP\_Update(): responsabila pentru actualizarea contoarelor din tabele
\end{itemize}

In cele ce urmeaza vom prezenta logica din spatele acestei interfete oferite de aplicatie.\\

\textbf{BP\_Init}\\
Funtie responsabila de initializarea parametrilor interni ai predictorului si anume:
\begin{itemize}
	\item bpTable: tablou ce inglobeaza toate tabelele predictorului
	\item aliasingCounter: contor folosit pentru a determina daca se va folosi istoria local sau globala
	\item theta\_threshold: variabila ce memoreaza pragul folosit pentru a decide daca contoarele din tabelele predictorului vor fi actualizate sau nu.
	\item ghr: serveste drept model pentru registrul GHR (Global History Register) 
\end{itemize}

\textbf{BP\_GetPrediction}
Functie ce are ca parametru de intrare valoarea curenta a registrului PC, valoare ce va fi folosita in selectia tabelelor corespunzatoare. Aceasta selectie se face pe baza aplicarii operatiei logic SAU EXCLUSIV sau XOR intre ce se afla in PC si indexul tabelelor ce contin contoarele (a se vedea apelul functiei BP\_GetCntIdx). Dupa cum s-a mentionat anterior, algoritmul mentioneaza nevoia de a calcula suma 
algebrica a contoarelor pentru a decide daca un salt se va face sau nu (da daca suma este pozitiva si nu altfel).

\textbf{BP\_Update}
Functie ce are ca parametri de intrare rezultatul predictiei (dacă se face sau nu saltul), rezultatul real, valoarea curenta a registrului PC împreuna cu valoarea adresei de salt. Aceasta funcție actualizeaza starea predictorului dupa ce predictia pentru salt a avut loc. Actualizăaile au loc daca predictorul a prezis incorect sau suma algebrica a ajuns sub nivelul pragului $\theta$ stabilit.
